# Mount
WinterTC is a standard for building HTTP servers behind Cloudflare, Deno, Vercel, and others. It enables web servers to operate across different runtimes using standard `Request` and `Response` APIs.

Elysia conforms to WinterTC standards, optimized for Bun while supporting additional runtimes. This interoperability allows multiple frameworks to work together seamlessly.

## Supported Frameworks

Compatible frameworks include:
- Hono
- Nitro
- H3
- Next.js API Routes
- Nuxt API Routes
- SvelteKit API Routes

Any framework that exposes a WinterTC-compliant `fetch` handler (`(request: Request) => Response | Promise<Response>`) can be mounted.

## Supported Runtimes

Deployment targets:
- Bun
- Deno
- Vercel Edge Runtime
- Cloudflare Worker
- Netlify Edge Function

## Basic Mount Pattern

Use `.mount()` to attach another framework's fetch handler at a specific path prefix:

```typescript
import { Elysia } from 'elysia'
import { Hono } from 'hono'

const hono = new Hono()
	.get('/', (c) => c.text('Hello from Hono!'))

const app = new Elysia()
    .get('/', () => 'Hello from Elysia')
    .mount('/hono', hono.fetch)
```

The `.mount()` method accepts a path and a `fetch` function conforming to the WinterTC standard (`(request: Request) => Response | Promise<Response>`).

## Nested Mounting

Frameworks can be nested arbitrarily deep:

```typescript
import { Elysia } from 'elysia'
import { Hono } from 'hono'

const elysia = new Elysia()
    .get('/', () => 'Hello from Elysia inside Hono inside Elysia')

const hono = new Hono()
    .get('/', (c) => c.text('Hello from Hono!'))
    .mount('/elysia', elysia.fetch)

const main = new Elysia()
    .get('/', () => 'Hello from Elysia')
    .mount('/hono', hono.fetch)
    .listen(3000)
```

This makes interoperable frameworks and runtimes a reality, allowing you to incrementally adopt Elysia or compose applications from multiple frameworks.

## `.mount()` vs `.use()` - Key Differences

| Feature | `.mount(path, fetchFn)` | `.use(plugin)` |
|---------|------------------------|----------------|
| **Input** | Any WinterTC `fetch` function | Elysia plugin instance |
| **Type safety** | No type propagation across boundary | Full Elysia type chain preserved |
| **Lifecycle hooks** | Elysia hooks do not run on mounted routes | Hooks apply normally |
| **Schema validation** | Handled by the mounted framework | Elysia validates via TypeBox/Zod etc. |
| **Eden support** | Mounted routes are not visible to Eden | Full Eden type inference |
| **Decorator/state** | Not shared with mounted framework | Shared through Elysia context |
| **Use case** | Integrating non-Elysia frameworks | Composing Elysia sub-applications |

In short: use `.use()` for Elysia plugins where you want type safety and lifecycle integration. Use `.mount()` when integrating external frameworks or legacy code that exposes a standard `fetch` handler.

## Performance Implications

- **Minimal overhead**: `.mount()` delegates directly to the mounted framework's `fetch` handler with only path prefix stripping. There is no middleware pipeline or lifecycle overhead from Elysia on mounted routes.
- **No double parsing**: The request is parsed once by whichever framework handles it. Elysia does not parse the body or query before delegating to a mounted handler.
- **Path matching cost**: Elysia checks the mount prefix on each request. With many mount points, this is a linear scan. For best performance, keep the number of top-level mounts reasonable.
- **Memory**: Each mounted framework maintains its own router and middleware stack in memory. Mounting many large framework instances increases memory usage.

## Limitations and Caveats

- **No Elysia lifecycle hooks**: `beforeHandle`, `afterHandle`, `onError`, and other Elysia lifecycle events do not run for requests handled by mounted frameworks. The mounted framework manages its own request lifecycle.
- **No type propagation**: Types from the mounted framework are not visible to Elysia or Eden. Mounted routes do not appear in OpenAPI/Swagger documentation generated by Elysia.
- **No shared state**: Elysia `store`, `decorate`, and `derive` values are not available inside mounted handlers. Data must be passed via request headers or other out-of-band mechanisms.
- **Path prefix stripping**: Elysia strips the mount prefix before forwarding. A request to `/hono/users` arrives at the Hono handler as `/users`. The mounted framework should define routes relative to its own root.
- **WebSocket**: WebSocket upgrade handling depends on the mounted framework's capabilities and the runtime. Not all combinations work seamlessly.

## Real-World Use Cases

### Incremental Migration
Gradually migrate a Hono or Express app to Elysia by mounting the legacy app and building new routes natively:

```typescript
import { Elysia } from 'elysia'
import { legacyApp } from './legacy-hono-app'

const app = new Elysia()
    // New routes with full Elysia features
    .get('/api/v2/users', () => fetchUsers())
    // Legacy routes continue working
    .mount('/api/v1', legacyApp.fetch)
    .listen(3000)
```

### Multi-Framework Microservices
Compose independently developed services into a single deployment:

```typescript
import { Elysia } from 'elysia'
import { authService } from './services/auth'     // Hono
import { cmsService } from './services/cms'        // Elysia plugin
import { analyticsService } from './services/analytics' // H3

const gateway = new Elysia()
    .use(cmsService)                               // Elysia plugin - full types
    .mount('/auth', authService.fetch)             // Hono - mounted
    .mount('/analytics', analyticsService.fetch)   // H3 - mounted
    .listen(3000)
```

### Serving a Static Site Framework
Mount a Next.js or SvelteKit app alongside an Elysia API:

```typescript
import { Elysia } from 'elysia'
import { nextHandler } from './next-handler'

const app = new Elysia()
    .group('/api', (app) =>
        app.get('/health', () => ({ status: 'ok' }))
    )
    .mount('/', nextHandler)
    .listen(3000)
```
